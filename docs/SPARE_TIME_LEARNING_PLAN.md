# 业余 Agent 工程师学习计划（工作之余版）

更新时间：2026-07-18

本文档把站点现有能力组织成一条 8 周业余学习主线，服务唯一目标：**成为能独立构建、面试和复盘的 Agent 开发工程师**。

## 前提与原则

- 时间预算：每周 6–8 小时（工作日 3 个晚上 × 1 小时 + 周末 1 个半天）。
- 串行不并行：同一时间只做一条主线，完成比开始重要。
- 证据先于表达：每一周都有可验证交付物和退出标准，没通过门禁就不进入下一周。
- 学完必自测：每个主题结束时做关联面试题（lab 页底部已挂"相关知识点 / 相关面试题"）。
- 复盘进 journal：每周至少让 Agent 保存 1 条学习笔记（`/agent` 运行时说"保存成学习笔记"，审批后在 `/journal` 查看）。

## 当前起点（2026-07-18）

已就绪的学习设施：4+1 个 labs（Prompt 回归、RAG 三路检索、受控 Agent、Agent 评测门禁、MCP 协议）、服务端 Agent runtime（真实工具、审批、回放、OTel 导出）、`/journal` 笔记闭环、lab 关联内容、CI 门禁。

唯一阻塞主线：Agent runtime 的真实模型基线（6.2B-2B），卡在账户所有者配置密钥。

## 第 1 周：真实模型主线（最高优先级）

对应面试能力域：模型接入可靠性、成本工程、发布门禁。

- 交付物：
  1. 在 Vercel 配置 `AGENT_RUNTIME_MODE=openai`、`OPENAI_API_KEY`（只存服务端环境，不进仓库/聊天记录/浏览器）。
  2. 重新部署后跑通 `npm run smoke:agent:live`（`releaseReady` + `liveModelReady`）。
  3. 跑 `npm run eval:agent:live` + `npm run verify:agent:live`，产出 20 条真实模型基线报告。
  4. 写一页失败样本分析：哪几条夹具真实模型做错了、为什么、怎么修。
- 退出标准：20/20 通过 + request id / Token / 成本覆盖率 100% + 独立复核通过；失败样本分析写进 journal。
- 学习重点：真实 planner 的工具选择可靠性、成本/延迟的量化方法、"夹具全过 ≠ 真实模型全过"的差距分析。这是面试里最有说服力的素材。

## 第 2 周：MCP 协议与工具契约

对应面试能力域：工具设计、协议理解。

- 交付物：
  1. 在 `/labs/mcp-tools` 完整跑 4 个预设会话，逐帧读懂 JSON-RPC 轨迹。
  2. 源码精读：`lib/agent-runtime/contracts.ts`、`tools.ts`、`mcp.ts`、`runner.ts`，画出"一次工具调用从 HTTP 到执行"的时序图（画在 journal 笔记里）。
  3. 用 curl 手工调用 `POST /api/mcp` 的 initialize / tools/list / tools/call，不看页面。
  4. 完成 lab 底部关联的 3 个 MCP 知识点 + 4 道面试题自测。
- 退出标准：能口头回答"为什么 save_learning_note 不经 MCP 暴露"和"MCP 与 function calling 什么关系"，各 3 分钟。
- 学习重点：协议层标准化 vs 模型能力层；最小权限在协议边界的落地。

## 第 3–4 周：检索质量深入（RAG 向量）

对应面试能力域：检索工程、评估驱动开发。

- 交付物：
  1. 在 `/labs/rag-retrieval` 跑三路模式（关键词/混合/向量）各 5 个自己的问题，记录差异。
  2. 新增 3 条"语义改写"评估题（换措辞、不共用关键词），展示向量模式真正优势的场景；重跑 `npm run build:rag-vectors` 与评估快照。
  3. 做一次阈值校准实验：分析 0.88 拒答阈值在新题集下是否仍成立，把校准过程写成 journal 笔记。
  4. 完成关联的 4 个 RAG 知识点 + 4 道面试题自测。
- 退出标准：评估题集从 13 条扩到 16 条且三路对比表更新；能讲清"为什么向量模式和混合检索在原题集打平"。
- 学习重点：评估集决定结论；小模型中文检索的真实边界；阈值校准方法论。

## 第 5 周：生产可观测性（OTel）

对应面试能力域：可观测性、生产运维。

- 交付物：
  1. 跑一次完整 Agent 运行，导出 OTel JSON，读懂 span 树和 `gen_ai.*` 属性。
  2. 本地用 Docker 起一个 Jaeger（或 Grafana Tempo），用官方 OTel SDK 给一个最小 demo 仪表化，对比 `lib/agent-runtime/otel.ts` 的映射差异。
  3. 完成关联的 tracing / evaluation 知识点自测。
- 退出标准：能讲清 trace/span/attribute 的层级关系，以及"为什么 token usage 只聚合在根 span"。
- 学习重点：标准可观测性是 prototype → production 的分水岭；自研 trace 与 OTel 的映射关系。

## 第 6 周：证据固化（把过程变成内容）

对应面试能力域：表达与复盘。

- 交付物：
  1. 把前 5 周的 journal 笔记整理成 1–2 篇知识点或学习日志（工程手册风格，有数字、有失败样本）。
  2. 按 `docs/PROJECT_EVIDENCE_AUDIT.md` 审计 Agent runtime 是否达到 `verified`，达到则走状态变更流程。
  3. 更新 `/career/calibration` 自测，对比 8 周前的基线。
- 退出标准：站点上至少 1 个项目状态变更或 1 篇新原创内容上线；自测分数变化有记录。

## 第 7–8 周：求职冲刺

- 交付物：
  1. 用 6 个真实 JD 校准证据覆盖，补齐排名前两的缺口。
  2. 每个 `prototype`/`verified` 项目演练 3 分钟讲解模板，录音自评。
  3. 完成每个岗位的固定 5 题模拟面试，四项量表自评。
- 退出标准：目标岗位证据覆盖率提升；3 分钟讲解不超时、有数字、有取舍说明。

## 不要做

- 不在真实模型基线完成前开新 lab。
- 不为凑数批量生成没有来源的内容。
- 不把 `prototype` 在简历里写成生产经验。
- 不把密钥交给任何对话框、仓库或浏览器端。
